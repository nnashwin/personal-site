The Right Way to Automate Testing

The concept of automation is not new to the software industry. Since there have been processes to test and flaky builds, there has been the need to define software that can test it and report the results to humans.

It was new to me when I worked at my first corporate job. Having done front-end development for a few years at multiple start ups, I was surprised that I hadn't even heard of the idea of automated UI testing.

After working on automation for a few months, it became evident that it was valuable, but also difficult to implement in a way that doesn't break due to extraneous issues every time someone wants to merge into master.

More recently, I have had the opportunity to help a couple of friends automation a project that has been easier to manage. We started out by defining a clear goal for the task we are automating. The smaller scale, focus on customer satisfaction (the person whose task is helping to automate) over coverage numbers, and bringing in an outside tester (me) to hammer the artifact has led to the development of a stable, healthier automation project.

Automation is just testing

Automation in the software industry is usually referring to either build automation, task automation, or test automation. The latter has become a trend in recent years (and maybe before, I don't have context here) as companies push for software teams to write code faster while trying to maintain quality. Automation testing of APIs and UI components is a hot topic, and many companies have jumped behind it with gusto.
